## Research Scope

* Question: Why are JavaScript files in public/common present in this TypeScript project, and are they required?
* Task 1: Find all references to files under public/common (HTML script tags, imports, server static serving, docs).
* Task 2: Inspect package.json and tsconfig.json to determine whether browser TypeScript is compiled to public/common or if these are hand-maintained JS runtime assets.
* Task 3: Determine if removing public/common/*.js would break currently served pages.
* Task 4: Provide concrete evidence with file paths and line numbers from workspace files.
* Task 5: Evaluate alternatives and tradeoffs: keep JS assets, migrate to TS + bundler/transpile step, or mixed approach.

### 2) Are these compiled TypeScript outputs or hand-maintained JS assets?

TypeScript compilation scope is server-side source only:

* tsconfig.json:7 sets rootDir to ./src.
* tsconfig.json:6 sets outDir to ./dist.
* tsconfig.json:26 includes only src/**/*.

Build scripts do not define any browser transpile/bundle pipeline to public/common:

* package.json:12 build script is only tsc.
* package.json:14 dev script runs tsx watch src/index.ts.

No TypeScript source files exist that map to these browser helpers:

* No matches under src for session-manager, colyseus-client, match-room-app, or lobby-slot-controls.

Runtime structure confirms hand-maintained browser JS globals:

* public/common/session-manager.js:287 exports global.KaijuSession.
* public/common/colyseus-client.js:382 exports global.KaijuColyseusClient.
* public/common/lobby-slot-controls.js:22 exports global.KaijuLobbySlots.
* public/common/match-room-app.js:2 reads global.KaijuSession.
* public/common/match-room-app.js:3 reads global.KaijuColyseusClient.

Conclusion: public/common/*.js are not generated by current TypeScript build config; they are committed runtime assets maintained directly as JavaScript.

### 3) Would removing public/common/*.js break currently served pages?

Yes, removing them would break current behavior immediately.

Direct breakage reasons:

* The server intentionally exposes /common from public/common (src/index.ts:48). If files are removed, /common/* requests return 404.
* Several pages hard-load these scripts with script tags (public/index.html:115, public/lobby.html:178-181, public/match-room.html:316-320, public/commander/index.html:117-118, public/kaiju/index.html:174-175).
* Entry page inline code directly calls window.KaijuSession.getPlayerName() (public/index.html:122). Without session-manager.js, this throws.
* Lobby page inline logic uses window.KaijuSession and window.KaijuColyseusClient (public/lobby.html:208, public/lobby.html:353, public/lobby.html:411, public/lobby.html:412, public/lobby.html:416).
* Commander app throws if helpers are missing (public/commander/app.js:665-667).
* Kaiju app throws if KaijuColyseusClient is missing (public/kaiju/app.js:1233-1235).

Test contract also codifies dependency on these paths:

* src/index.test.ts:231-263 expects /common assets to be present and served.

Note on test execution in current workspace state:

* A focused test run was executed (npm test -- --runInBand src/index.test.ts), but the suite currently fails broadly in this environment for unrelated setup/runtime reasons. Even so, the static assertions listed above still document intended contract and routing expectations.

### 4) Why these files are present in a TypeScript project

Repository architecture is mixed:

* Backend is TypeScript compiled from src to dist.
* Browser clients are plain JS/HTML in public, loaded directly by the browser without a bundler step.
* public/common contains shared browser helper modules implemented as global objects and reused by multiple pages.

This is a valid architecture, but it is not TypeScript end-to-end.

### 5) Alternatives and tradeoffs

Option A: Keep public/common as JavaScript runtime assets (status quo)

* Benefits:
	* Zero migration cost.
	* No build-step changes for browser code.
	* Easy to debug in browser as plain served files.
* Costs:
	* No TypeScript type safety in browser helper layer.
	* Global namespace coupling (window.KaijuSession, window.KaijuColyseusClient).
	* Harder refactoring and weaker compile-time guarantees.

Option B: Migrate browser helpers to TypeScript + bundler/transpile pipeline

* Approach:
	* Move common helpers (and optionally page apps) into src/client or similar.
	* Add explicit build step (esbuild, Vite, or similar) that emits assets to public or dist/public.
	* Replace global contracts with ES module imports where practical.
* Benefits:
	* Type safety and stronger refactoring support.
	* Better dependency management and dead-code elimination options.
	* More maintainable long-term client architecture.
* Costs:
	* Migration complexity and risk.
	* Build/serve pipeline changes required.
	* Potential churn in script loading order and deployment packaging.

Option C: Mixed approach (recommended incremental path)

* Approach:
	* Keep current HTML and server static wiring initially.
	* Convert public/common one file at a time to TS source, transpile to the same output filenames, preserve global API shape during transition.
	* Add lightweight client build script while keeping backend tsc build unchanged.
* Benefits:
	* Low-risk migration with backward compatibility.
	* Immediate type-safety gains in highest-value shared helpers.
	* Avoids full frontend rewrite.
* Costs:
	* Temporary dual-mode complexity.
	* Need discipline to avoid divergence between generated and hand-edited outputs.

## Explicit Verdict

### Do we really need these files right now?

Yes. In the current architecture, public/common/*.js are required runtime assets. They are directly loaded by served pages, relied on by inline and page-level client logic, statically mapped by the Express server, and referenced by tests as part of expected application behavior.

You can remove them only if you simultaneously replace their runtime contract with an alternative client build/output path and update page wiring accordingly.
